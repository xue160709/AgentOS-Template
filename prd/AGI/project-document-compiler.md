# AgentOS 文档驱动智能体系统生成 PRD

## 一、产品背景

很多项目并不缺文档，真正的问题是：文档没有变成 Agent 可以持续使用的工作系统。

一个成熟项目里通常已经有 README、PRD、架构说明、接口文档、设计规范、测试规范、发布流程、会议纪要、TODO、Roadmap、历史决策和复盘记录。但当前大多数 Agent 使用这些文档的方式仍然很粗糙：

- 用户每次都要重新解释项目背景。
- Agent 读了文档，但不知道哪些文档更权威。
- 文档里的流程、规范和验收标准没有变成可复用能力。
- 项目里的角色分工没有变成可调度的 Subagents。
- 运行后的经验没有沉淀回长期记忆。
- 文档只是在被搜索，而不是在驱动 AgentOS 的行为。

AgentOS 需要把“项目文档”从静态资料升级为“智能体系统的来源”。用户添加项目后，AgentOS 应该能够基于项目文档自动生成一套项目专属的 Agent 工作系统。

## 二、产品定位

本功能是 AgentOS 的“项目文档智能体化”能力。

它不是一个普通文档搜索功能，也不是把所有文档塞进一个大 Prompt。它的核心目标是：

> 把项目文档转化为 AgentOS 可运行、可审阅、可演进的 Skills、Subagents、Guardrails、Workflows、Project Memory 和 Task Graph。

换句话说，项目文档应该成为 AgentOS 的“源码”。

```text
项目文档
  -> 项目世界模型
  -> 项目记忆
  -> Skills
  -> Subagents
  -> Guardrails
  -> Workflows
  -> Task Graph
  -> 项目专属 AgentOS
```

## 三、产品目标

### 主要目标

1. 用户添加项目后，AgentOS 能快速理解项目背景、目标、模块、约束和当前任务。
2. AgentOS 能从文档中提取可复用工作方式，生成项目专属 Skills。
3. AgentOS 能根据项目结构和文档职责，生成项目专属 Subagents。
4. AgentOS 能从规范、流程和历史经验中提取 Guardrails。
5. AgentOS 能将 PRD、TODO、Roadmap 和历史记录转化为可执行 Task Graph。
6. 用户可以审阅、编辑、启用或停用生成结果。
7. Agent 执行任务时，可以按任务需要选择相关文档、Skills、Subagents 和 Guardrails。
8. Agent 执行后的稳定经验可以作为候选记忆被沉淀，而不是丢在一次性对话里。

### 非目标

1. 不追求“自动实现 AGI”。
2. 不训练一个新的模型。
3. 不把所有文档无差别注入每次对话。
4. 不让 Agent 在用户未确认的情况下重写原始文档。
5. 不把短期运行日志直接当成长期记忆。
6. 不要求一开始就实现复杂企业知识库、权限系统或跨项目协同。

## 四、核心用户

### 项目创建者

已经有一个项目目录，里面有大量文档，希望 AgentOS 添加项目后就能理解项目，不需要从零解释。

### 产品负责人

希望 PRD、用户场景、验收标准、Roadmap 不只是给人看的，也能成为 Agent 拆任务、判断优先级和验收结果的依据。

### 开发者

希望 Agent 修改代码、生成方案或排查问题时，能自动遵守项目架构、代码规范、测试要求和历史决策。

### 设计与体验负责人

希望设计规范、组件规则、交互原则和视觉约束能变成 UI 相关任务的默认 Guardrails。

### AgentOS 高级用户

希望把一个项目逐渐训练成“有项目记忆、有工作流、有角色分工、有自我校验”的长期智能工作空间。

## 五、核心概念

### 1. 文档是世界模型

项目文档不只是知识来源，而是 AgentOS 对项目世界的结构化理解来源。

AgentOS 需要从文档中回答：

- 这个项目是做什么的？
- 用户是谁？
- 核心场景是什么？
- 核心业务对象是什么？
- 项目有哪些模块？
- 当前优先级是什么？
- 哪些文档是权威来源？
- 哪些规则不能被违反？
- 哪些问题还不清楚？

最终产物是 Project Knowledge Map。

### 2. Project Memory 是长期稳定记忆

Project Memory 不是聊天记录，也不是文档全文摘要。

它应该是一份简洁、稳定、可审阅的项目长期记忆，记录 Agent 在任何任务中都应该知道的背景：

- 产品目标。
- 用户画像。
- 核心概念。
- 核心场景。
- 架构概览。
- 业务规则。
- 技术约束。
- 设计原则。
- 测试要求。
- 当前优先级。
- 权威文档列表。

Project Memory 的原则是少而稳定。只有经过确认、去重、长期有效的信息才应该进入这里。

### 3. Skills 是文档中的能力插件

项目文档里经常隐藏着大量可复用流程。

例如：

- “新增接口时必须同步更新 API 文档和测试。”
- “UI 页面必须包含加载、空状态、错误状态。”
- “发布前需要完成回归测试和版本说明。”
- “修改数据模型时必须检查兼容性。”
- “PRD 变更后要同步更新验收标准。”

这些内容不应该只停留在文档里，而应该转化为 Skills。

Skill 的本质是：

```text
触发条件 + 工作流程 + 所需上下文 + 检查标准 + 禁止事项
```

也就是说，Skill 不是一段提示词，而是项目中某类任务的可复用操作规程。

### 4. Subagents 是项目角色分工

Subagents 不应该凭空生成，而应该从项目文档、模块边界和工作方式中推导出来。

一个项目可能需要：

- Orchestrator：理解目标、拆任务、选择 Skills 和 Subagents。
- Product Agent：理解 PRD、用户场景、验收标准。
- Research Agent：查找相关文档、事实和背景。
- Frontend Agent：处理页面、组件、交互和设计规范。
- Backend Agent：处理接口、数据模型、服务逻辑。
- QA Agent：生成测试计划、边界场景和回归检查。
- Docs Agent：同步 PRD、架构文档、变更说明。
- Review Agent：发现风险、遗漏、规范冲突和质量问题。

关键不是“Agent 越多越好”，而是每个 Subagent 必须有清晰职责、输入范围和输出标准。

### 5. Guardrails 是项目边界

Guardrails 来自文档里的规则、限制、风险和价值观。

例如：

- 不能绕过权限系统。
- 不能擅自删除历史数据。
- 涉及用户隐私时必须检查数据暴露风险。
- 修改接口必须保持兼容或提供迁移说明。
- UI 变更必须考虑移动端。
- 关键流程必须有测试。
- 不明确的产品决策必须向用户确认。

Guardrails 的作用不是让 Agent 变慢，而是让 Agent 知道哪些地方必须谨慎、验证或请求确认。

### 6. Workflows 是任务执行路径

不同类型任务应该有不同工作流。

例如：

```text
Feature Workflow:
理解需求 -> 查相关文档 -> 拆子任务 -> 执行 -> 验证 -> Review -> 文档同步 -> 记忆候选
```

```text
Bugfix Workflow:
复现问题 -> 定位原因 -> 查历史记录 -> 修复 -> 回归测试 -> 记录风险
```

```text
Docs Update Workflow:
识别变更 -> 找相关文档 -> 更新内容 -> 检查一致性 -> 标记待确认问题
```

Workflow 把 Agent 从“直接回答”变成“按照项目期望工作”。

### 7. Task Graph 是文档中的目标系统

PRD、TODO、Roadmap、Issue、会议纪要和复盘中，往往已经存在很多任务线索。

AgentOS 应该从这些文档中抽取：

- 待实现功能。
- 已知 Bug。
- 重构事项。
- 测试缺口。
- 文档缺口。
- 发布准备。
- 未决问题。
- 任务依赖。
- 验收标准。

Task Graph 的价值是让 AgentOS 知道项目“接下来可以做什么”，而不是只在用户输入任务时被动响应。

## 六、用户体验流程

### 流程一：添加项目

用户把一个项目添加到 AgentOS。

AgentOS 首先不是直接进入聊天，而是展示一个项目理解过程：

```text
正在理解项目：
- 发现了哪些文档
- 文档大致属于哪些类型
- 哪些文档可能是权威来源
- 可以生成哪些 Skills
- 可以生成哪些 Subagents
- 有哪些不确定问题需要用户确认
```

### 流程二：生成项目智能体系统

AgentOS 基于项目文档生成一套草案：

- Project Knowledge Map。
- Project Memory。
- Skills。
- Subagents。
- Guardrails。
- Workflows。
- Task Graph。
- Open Questions。

这些内容默认是草案，不直接变成不可见的系统提示。

### 流程三：用户审阅

用户可以看到 AgentOS 是如何理解项目的。

用户可以：

- 接受全部生成结果。
- 只接受部分 Skills。
- 修改某个 Subagent 的职责。
- 关闭某些 Guardrails。
- 标记某篇文档为权威来源。
- 回答系统提出的开放问题。
- 要求重新生成某一部分。

### 流程四：执行任务

当用户输入任务，例如：

```text
帮我做一个新的订单详情页
```

AgentOS 应该自动判断：

- 这是一个 Feature 任务。
- 需要 Product Agent 理解订单相关 PRD。
- 需要 Frontend Agent 处理页面和组件。
- 需要 UI Skill 和 Docs Sync Skill。
- 需要读取设计规范、订单业务文档和前端架构文档。
- 需要 Review Agent 检查交互、边界状态和测试。

用户应该能看到本次任务使用了哪些上下文和角色，而不是完全黑盒。

### 流程五：验证与反馈

任务完成后，AgentOS 不只输出结果，还应该进行总结：

- 完成了什么。
- 使用了哪些 Skills。
- 哪些文档被引用。
- 哪些风险仍然存在。
- 哪些测试或验证已完成。
- 哪些信息值得写入长期记忆。
- 哪些文档可能需要更新。

需要沉淀的内容进入 Memory Update Candidate，由用户确认后再写入长期记忆。

## 七、生成内容说明

### 1. Project Knowledge Map

Project Knowledge Map 是项目的结构化理解层。

它应该包含：

- 项目一句话说明。
- 核心用户。
- 核心场景。
- 核心业务概念。
- 模块划分。
- 文档分类。
- 权威来源。
- 文档之间的关系。
- 已知冲突。
- 开放问题。

它解决的问题是：AgentOS 知道这个项目是什么，以及知识在哪里。

### 2. Project Memory

Project Memory 是项目级长期记忆。

它应该包含：

- 产品目标。
- 用户画像。
- 核心概念。
- 业务规则。
- 架构摘要。
- 重要约束。
- 当前优先级。
- 测试和质量要求。
- 权威文档列表。

它解决的问题是：AgentOS 在每次任务前不需要重新理解项目基本背景。

### 3. Skills

Skills 是从项目文档中提取的可复用能力。

常见 Skills 包括：

- 需求拆解 Skill。
- UI 实现 Skill。
- API 变更 Skill。
- 测试补全 Skill。
- 文档同步 Skill。
- 发布检查 Skill。
- 架构评审 Skill。
- Bug 定位 Skill。

每个 Skill 应该说明：

- 什么情况下使用。
- 需要读取哪些上下文。
- 推荐工作步骤。
- 需要检查哪些结果。
- 有哪些不能违反的规则。
- 来源文档是什么。

### 4. Subagents

Subagents 是根据项目工作方式生成的角色。

每个 Subagent 应该说明：

- 它负责什么。
- 它不负责什么。
- 默认应该看哪些文档。
- 可以使用哪些 Skills。
- 输出应该包含什么。
- 什么情况下需要交给其他 Agent 或用户。

Subagent 的重点是职责清晰，而不是数量多。

### 5. Guardrails

Guardrails 是项目执行边界。

它应该包含：

- 安全边界。
- 权限边界。
- 数据边界。
- 架构边界。
- 设计边界。
- 测试边界。
- 文档边界。
- 人工确认边界。

Guardrails 需要有来源。用户应该知道某条 Guardrail 是从哪篇文档、哪条规范或哪段历史经验来的。

### 6. Workflows

Workflows 是项目任务的默认执行路径。

常见 Workflows 包括：

- 新功能。
- 修 Bug。
- 重构。
- 文档更新。
- 发布准备。
- 技术调研。
- 设计评审。
- 测试补全。

Workflow 需要定义：

- 任务开始时先理解什么。
- 需要哪些角色参与。
- 需要调用哪些 Skills。
- 中间结果如何验证。
- 完成后是否需要更新文档或记忆。

### 7. Task Graph

Task Graph 是从项目文档中抽取的任务网络。

它应该包含：

- 任务标题。
- 任务类型。
- 来源文档。
- 相关背景。
- 验收标准。
- 依赖关系。
- 推荐 Skills。
- 推荐 Subagents。
- 当前状态。

Task Graph 让 AgentOS 从“聊天工具”变成“项目推进系统”。

### 8. Open Questions

Open Questions 是系统不能确定、需要用户判断的问题。

例如：

- 两份 PRD 对同一功能描述不一致。
- README 和架构文档描述的技术栈不同。
- TODO 中的任务没有验收标准。
- 某个模块缺少负责人或说明。
- 某条规则看起来已经过时。

Open Questions 很重要，因为它避免 AgentOS 假装理解。

## 八、核心功能需求

### P0：项目文档理解

AgentOS 添加项目后，需要能识别并分类项目文档。

分类至少包括：

- 产品文档。
- 技术文档。
- 流程文档。
- 设计文档。
- 测试文档。
- 决策文档。
- 任务文档。
- 记忆文档。
- 领域知识。

输出结果应该让用户知道：AgentOS 读到了什么，以及它如何理解这些文档。

### P0：Project Memory 生成

AgentOS 需要生成一份简洁的项目长期记忆。

这份记忆不能是所有文档的摘要堆叠，而应该是稳定、必要、可复用的项目背景。

### P0：Guardrails 生成

AgentOS 需要从文档中提取项目边界。

Guardrails 必须可追溯，并且可以被用户编辑。

### P0：Workflow 生成

AgentOS 需要基于项目文档生成基本任务工作流。

至少包括：

- 新功能工作流。
- Bug 修复工作流。
- 文档更新工作流。
- Review 工作流。

### P1：Skills 生成

AgentOS 需要从流程、规范和经验中生成 Skills。

Skills 应该是面向任务的，而不是面向文档的。也就是说，它回答的是“遇到这类事情该怎么做”，而不是“这篇文档说了什么”。

### P1：Subagents 生成

AgentOS 需要根据项目结构和工作方式生成 Subagents。

默认可以生成基础角色，但用户应该能删除、合并或改名。

### P1：Task Graph 生成

AgentOS 需要从 PRD、TODO、Roadmap、Issue、会议纪要中抽取任务。

任务需要保留来源，并尽可能关联验收标准和依赖。

### P1：审阅和编辑

所有生成内容都应该可审阅。

用户可以：

- 接受。
- 修改。
- 停用。
- 重新生成。
- 标记为过时。
- 标记为权威。

### P2：运行反馈回写

Agent 完成任务后，系统可以生成记忆更新候选。

但回写长期记忆必须谨慎：

- 只沉淀长期有效信息。
- 不记录临时失败噪声。
- 不记录无意义过程日志。
- 用户可以确认或拒绝。

### P2：文档冲突检测

AgentOS 应该能识别文档之间的明显冲突，并以问题形式呈现给用户。

冲突不应该被系统擅自解决，除非已有明确权威来源。

## 九、产品原则

### 1. 文档不是背景材料，而是系统来源

AgentOS 不只是“能搜索文档”，而是要把文档变成系统能力、角色、流程和边界。

### 2. 生成结果必须可见

用户应该能看到 AgentOS 如何理解项目，而不是只能相信一个黑盒。

### 3. 长期记忆必须克制

不是所有信息都值得记住。长期记忆应该稳定、少量、经过确认。

### 4. Skills 应该来自真实流程

Skill 不应该凭空生成。它应该来自项目文档中的规则、流程、检查项和历史经验。

### 5. Subagents 应该服务于任务，而不是制造复杂性

多 Agent 的价值在于职责清晰、并行处理和质量检查，而不是让系统看起来更复杂。

### 6. 上下文应该按需选择

AgentOS 不应该每次把所有文档都塞给 Agent。它应该根据任务选择相关文档、Skills、Subagents 和 Guardrails。

### 7. 不确定性必须显式暴露

当文档冲突、缺失或过期时，系统应该提出问题，而不是编造确定答案。

## 十、与 2023 AGI 框架的对应关系

| 2023 框架概念 | AgentOS 中的对应能力 |
| --- | --- |
| 任务、目标等数据 | Task Graph、用户输入目标、项目 Roadmap |
| 短时记忆 | 当前任务上下文、对话线程、执行过程 |
| 长时记忆 | Project Memory、项目文档、Skills、历史决策 |
| RNN + Transformer 预测能力 | Planner、Evaluator、任务优先级判断、风险预测 |
| LLM Prompt 元认知 | Orchestrator 的任务理解、反思、自检和策略选择 |
| 执行 | Subagents、工具调用、任务工作流 |
| Guardrails | 项目边界、安全规则、质量标准、人工确认点 |
| 管理角色 | Orchestrator / Manager Agent |
| 执行角色 | Product、Frontend、Backend、QA、Docs、Review 等 Subagents |
| 对话面板 | 项目控制台、任务进度、文档引用、风险展示 |
| 外部信息 | 项目文档、代码、网页、知识库、历史记录 |
| 输出结果回写 | Memory Update Candidate、文档同步、经验沉淀 |

这说明，早期框架中“模型内部”的很多能力，在今天可以外化为可审阅、可编辑、可组合的 AgentOS 组件。

## 十一、示例场景

### 场景：用户添加一个已有项目

项目中有：

- README。
- PRD。
- 设计规范。
- 前端架构文档。
- 后端接口文档。
- TODO。
- 发布流程。

AgentOS 生成：

- Project Memory：项目目标、用户、核心模块和约束。
- Knowledge Map：文档分类、权威来源和开放问题。
- Skills：UI 实现、API 变更、测试补全、文档同步、发布检查。
- Subagents：Product Agent、Frontend Agent、Backend Agent、QA Agent、Docs Agent、Review Agent。
- Guardrails：权限、数据兼容、移动端 UI、测试和文档同步规则。
- Workflows：新功能、Bugfix、重构、发布。
- Task Graph：从 TODO 和 PRD 中提取的候选任务。

### 场景：用户要求新增功能

用户输入：

```text
帮我做订单详情页
```

AgentOS 判断：

- 任务类型：新功能。
- 相关文档：订单 PRD、设计规范、前端架构、API 文档。
- 需要 Skills：UI 实现、API 对接、测试补全、文档同步。
- 需要 Subagents：Product Agent、Frontend Agent、QA Agent、Review Agent。
- 需要 Guardrails：移动端适配、错误状态、接口兼容、测试要求。

最终输出不只是代码或回答，而是一条完整任务链：

```text
需求理解 -> 方案拆解 -> 实现 -> 验证 -> Review -> 文档同步 -> 记忆候选
```

### 场景：系统发现文档冲突

AgentOS 发现：

- PRD 说订单详情页需要展示优惠券。
- 设计稿说明中没有优惠券区域。
- API 文档没有优惠券字段。

AgentOS 不应该擅自决定，而应该生成 Open Question：

```text
订单详情页是否需要展示优惠券？
来源冲突：
- PRD 要求展示
- 设计文档未体现
- API 文档缺少字段
建议由用户确认后再进入实现
```

## 十二、成功指标

### 用户价值指标

- 用户添加项目后，需要手动解释项目背景的次数减少。
- 用户更容易知道 AgentOS 理解了哪些文档。
- 用户能接受或编辑生成的 Skills 和 Subagents。
- Agent 执行任务时更少遗漏项目规范。
- Agent 输出更经常附带文档来源和验证结果。

### 系统质量指标

- 生成的 Project Memory 简洁且长期有效。
- Skills 能真实反映项目流程。
- Subagents 职责不重叠或少重叠。
- Guardrails 能覆盖高风险边界。
- Workflows 能匹配常见任务类型。
- Open Questions 能暴露真实不确定性。

### 长期指标

- 项目使用越久，AgentOS 对项目越稳定。
- 文档变更能逐渐反映到 Skills、Memory 和 Workflows。
- 用户可以把 AgentOS 当成项目长期协作系统，而不是一次性聊天工具。

## 十三、优先级规划

### 第一阶段：文档到项目理解

目标是让 AgentOS 能读懂项目。

包含：

- 文档分类。
- Knowledge Map。
- Project Memory。
- 基础 Guardrails。
- Open Questions。

### 第二阶段：项目理解到工作系统

目标是让 AgentOS 能根据项目文档生成工作方式。

包含：

- Skills。
- Subagents。
- Workflows。
- Task Graph。
- 用户审阅和编辑。

### 第三阶段：工作系统到持续演进

目标是让 AgentOS 在长期使用中变得更贴合项目。

包含：

- 运行反馈回写。
- 文档冲突检测。
- 增量更新。
- 任务完成后的文档同步建议。
- 项目级经验沉淀。

## 十四、关键风险

### 1. 文档质量不稳定

项目文档可能过时、冲突或缺失。

应对方式：生成 Open Questions，不假装确定。

### 2. 生成内容过多

如果一开始生成太多 Skills 和 Subagents，用户会难以理解。

应对方式：默认生成少量高置信度结果，其余作为建议。

### 3. 长期记忆污染

如果把所有执行过程都写入记忆，项目记忆会变得混乱。

应对方式：只生成记忆候选，由用户确认后沉淀。

### 4. 多 Agent 复杂度失控

Subagents 过多会让系统变慢、变贵、变难解释。

应对方式：以任务需要为中心，默认少角色，允许用户扩展。

### 5. 文档和实际代码不一致

文档可能说一套，代码实际做一套。

应对方式：标记为风险，不直接把文档当绝对真理。

## 十五、开放问题

1. 项目首次导入时，默认应该生成多少个 Skills 才不会压迫用户？
2. Subagents 是按通用角色生成，还是按项目模块生成？
3. 用户是否需要一个“AgentOS 系统图”，可视化展示 Memory、Skills、Subagents、Workflows 的关系？
4. Project Memory 是否应该有“稳定记忆”和“近期重点”两个层级？
5. Open Questions 应该在项目导入时集中展示，还是在相关任务触发时再展示？
6. Task Graph 是否应该成为项目首页的核心视图？
7. 如何让用户快速判断某个 Skill 是可信的、过时的，还是需要修改？

## 十六、一句话总结

AgentOS 添加项目后，不应该只是“读取项目文档”，而应该把项目文档转化为一套可见、可编辑、可运行、可持续演进的智能体工作系统。

文档是世界模型，Skills 是能力插件，Subagents 是角色组织，Guardrails 是边界，Workflows 是执行路径，Task Graph 是目标系统，Project Memory 是长期记忆。
