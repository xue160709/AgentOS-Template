# 记忆系统方案建议

## 结论

目前更适合我们项目的方向不是简单复制某一个系统，而是做一个 **用户可治理的记忆系统**。

推荐方案是：以 **OpenClaw 的文件型记忆系统** 作为可读可改的主干，吸收 **Hermes 的 USER.md 用户画像层**，再把 **MemPalace 的原文抽屉/证据库** 作为增强层。

如果只能三选一，建议优先选择 OpenClaw。

核心原因是：我们的要求不只是“记得多”，而是“用户可以理解、可以修改、可以和记忆系统交互”。OpenClaw 的核心记忆是 Markdown 文件，例如 `MEMORY.md` 和 `memory/*.md`，用户天然可以打开、阅读、删除、修改和整理。它比纯向量库或数据库型记忆更透明，也更容易让用户建立信任。

但这不是完整答案。OpenClaw 解决了“记忆可读可改”的底层形态，却没有完整解决“用户如何治理记忆”。所以我们的方案需要在存储层之上增加一层明确的记忆治理机制：确认、修改、删除、冲突处理、禁记规则、过期和来源追溯。

一句话：我们要做的不是一个隐藏在后台的记忆库，而是一个用户可以共同维护的记忆关系系统。

## 三个方案的主要缺点

### OpenClaw 的缺点

OpenClaw 最适合作为底座，但它原本更偏“项目记忆”，不是天然的人类画像系统。它很适合记录项目规则、架构决策、工作流和每日沉淀，但对“用户是谁、用户偏好什么、用户和某个人或团队的关系”这类人的记忆，需要额外补 `USER.md`、`memory/people/*.md` 等设计。

另一个问题是自动沉淀机制较强。compaction flush、session-memory hook、dreaming promotion 都能自动整理记忆，这对工程系统很有价值，但对用户体验来说可能产生“不知道系统什么时候改了记忆”的不安。因此，OpenClaw 式自动写入不能直接照搬为默认长期记忆写入，必须增加待确认、来源解释和撤销能力。

还有一个长期维护问题：Markdown 文件越多，越容易重复、过期、冲突和失控。如果没有整理机制和 UI，用户一开始会觉得透明，后面会觉得难维护。

### Hermes 的缺点

Hermes 的优势是清晰地区分了两类核心记忆：

- `MEMORY.md`：环境事实、项目约定、工具经验、工作流教训。
- `USER.md`：用户身份、偏好、沟通风格、明确纠正。

这对“记忆不仅对产品，也对人”非常重要。

但 Hermes 的内置记忆偏短小、偏助手自身运行时。它更像一个随身小本子，不太像完整的项目记忆系统。外部 provider 虽然强大，但用户未必能直接理解和修改里面的内容。

Hermes 还有一个体验上的取舍：它为了保护 prefix cache，会使用冻结的系统提示词快照。记忆写入后，不一定马上反映在当前 session 的系统提示词里。技术上合理，但用户可能会困惑：“我刚让你记住，为什么你马上还没用上？”

所以 Hermes 更适合贡献“人类画像层”的设计，而不是单独作为我们的全部记忆底座。

### MemPalace 的缺点

MemPalace 很强，尤其适合大量历史资料、聊天记录、文档、项目文件的长期检索。它把原文 chunk 存成 drawer，把索引 pointer 存成 closet，还可以用知识图谱记录实体和时态事实。

但它的问题是：对普通用户来说，ChromaDB、drawer、closet、KG 都比较像系统内部结构。用户如果想直接理解和修改记忆，需要额外做一层 UI 或导出层。

另外，MemPalace 更擅长“找回原文”，不等于自动形成干净、可编辑、长期有效的人类记忆。它能告诉我们历史资料在哪里，但“这条信息是否应该变成长期记忆”“是否还有效”“是否需要用户确认”，仍然需要上层治理。

所以 MemPalace 更适合做后期的“大规模检索和证据层”，不建议一开始作为用户可编辑记忆的主系统。

## 推荐的整体结构

建议采用五层结构：

```text
第一层：用户可见的权威记忆
第二层：AI 自动建议和待确认记忆
第三层：原文证据库
第四层：检索和结构化理解层
第五层：用户治理层和记忆 UI
```

### 第一层：用户可见的权威记忆

这是整个系统的核心。重要记忆最终都应该能落到 Markdown 文件或等价的 UI 卡片里。

建议文件结构：

```text
MEMORY.md              产品/项目长期记忆：规则、架构、决策、工作流
USER.md                人的长期记忆：偏好、沟通方式、目标、禁忌、背景
memory/daily/*.md      每日/会话沉淀
memory/people/*.md     具体人的画像和关系
memory/projects/*.md   产品、功能、客户、场景记忆
```

这层必须具备几个特点：

- 用户能直接看懂。
- 用户能直接修改。
- 每条重要记忆最好有来源。
- AI 不能把用户明确否定的内容反复写回来。
- 记忆需要区分产品事实和人的事实。
- 关于人的记忆默认更谨慎。

### 第二层：AI 自动建议和待确认记忆

AI 不应该默认悄悄改写长期记忆。更好的方式是：

1. AI 发现可能值得保存的内容。
2. 先放入“待确认记忆”。
3. 用户确认、修改或拒绝。
4. 确认后再写入 `MEMORY.md`、`USER.md` 或对应分类文件。

例如：

```text
AI 建议记住：
- 用户偏好简洁直接的结论。
- 当前项目的记忆系统需要用户可理解、可修改。
- 产品记忆和人的记忆要分开管理。

操作：
[保存] [修改后保存] [忽略] [永不记录类似内容]
```

这会让记忆系统更像一个可以协作的对象，而不是一个偷偷学习用户的黑箱。

### 第三层：原文证据库

这一层可以吸收 MemPalace 的思路。

它不一定直接面向用户编辑，而是负责保存：

- 历史聊天原文。
- 项目文件片段。
- 文档内容。
- 用户确认过的记忆来源。

它的作用是回答：

- “这条记忆从哪里来的？”
- “当时我们到底怎么说的？”
- “有没有相关的历史上下文？”

证据库不应该成为唯一权威事实源。真正被系统长期采纳的事实，仍应沉淀到第一层的可编辑记忆里。

### 第四层：检索和结构化理解层

这一层可以使用向量检索、关键词检索、BM25、知识图谱、实体关系和时间线。

它的职责不是直接决定“什么是真记忆”，而是帮助系统找到相关材料、发现冲突、辅助归纳和提供证据。

可以吸收 MemPalace 的三个思路：

- 原文 chunk 是底线，回答重要问题时要能回到原文。
- 索引和摘要只是辅助，不能替代原文。
- 知识图谱适合记录会变化的关系和时态事实。

### 第五层：用户治理层和记忆 UI

只用 Markdown 文件还不够。为了让用户真的能理解和管理记忆，应该提供一个记忆面板。

建议包含这些视图：

- **关于我**：用户身份、偏好、长期目标、沟通方式。
- **关于产品**：项目规则、架构决策、功能背景、用户画像。
- **关于关系**：我和某人、某客户、某团队的上下文。
- **待确认**：AI 建议保存但还没确认的记忆。
- **冲突**：新信息和旧记忆不一致时，让用户选择保留哪一个。
- **来源**：每条记忆可以点开，看到来自哪次对话、哪个文档或哪个项目文件。
- **禁记规则**：用户可以声明哪些内容不要记录。

治理层需要支持这些动作：

- 确认：把待确认记忆变成正式记忆。
- 修改：用户可以改写 AI 建议的记忆。
- 删除：删除错误或不想保留的记忆。
- 过期：标记某条记忆已经不适用。
- 归档：保留历史但不主动使用。
- 冲突处理：当新旧记忆矛盾时，让用户选择。
- 禁记：用户可以规定某类内容永远不要记录。

建议每条记忆都带状态：

```text
pending     待确认
confirmed   已确认
stale       已过期
conflicted  有冲突
archived    已归档
blocked     禁止记录
```

记忆卡片示例：

```text
记忆：用户喜欢先听结论，再听理由。
类型：沟通偏好
对象：用户
来源：2026-05-28 对话
状态：已确认
操作：编辑 / 删除 / 设为过期 / 禁止类似记忆
```

## 最小可行版本

如果先做 MVP，建议先做下面这些：

1. 建立 `MEMORY.md` 和 `USER.md` 两个权威文件。
2. 让 AI 能搜索和读取这两个文件。
3. 增加“待确认记忆”文件或表。
4. 让 AI 在合适时提出“是否保存这条记忆”。
5. 用户确认后再写入 Markdown。
6. 每条保存的记忆附带类型、来源、时间和状态。
7. 做一个最小记忆面板：查看、编辑、删除、确认、禁记。

这个版本不需要一开始就做复杂知识图谱，也不需要立刻做大型向量库。先把“用户知道系统记了什么，并且能改掉记错的东西”做好。

## 特别注意

### 不要默认自动写长期记忆

AI 可以自动提出建议，但长期记忆最好默认进入“待确认”。否则用户会觉得系统在背后替他定义自己。

产品事实可以适度自动化，例如“这个项目使用某个架构”。人的记忆要更谨慎，例如身份、偏好、关系、情绪、健康、财务、价值观等，默认应要求用户确认。

### Markdown 是底层，不是唯一入口

Markdown 很适合透明存储，但普通用户最好面对记忆卡片和记忆面板。文件可以作为底层可导出、可审计格式，UI 才是主要交互入口。

### 记忆需要定期整理

记忆不能无限增长。系统需要定期合并重复项、标记过期项、发现冲突项，并把低价值历史归档。否则可编辑记忆会慢慢变成没人敢动的杂乱笔记。

### 证据和记忆要分开

原文证据不是长期记忆本身。证据回答“信息从哪里来”，记忆回答“系统以后应该如何理解和行动”。两者有关联，但不应该混为一谈。

## 设计原则

记忆系统最重要的不是聪明，而是可信。

因此，我们应该优先保证：

- 用户知道系统记了什么。
- 用户能修改或删除错误记忆。
- 用户能规定什么不要记。
- 记忆可以区分“关于产品”和“关于人”。
- AI 只能建议，关键长期记忆需要可审核。
- 每条重要记忆最好能追溯来源。
- 人的记忆比产品记忆更敏感，默认需要更高确认门槛。
- 系统可以帮助整理记忆，但不能替用户决定他们是谁。

最终目标不是做一个隐藏在后台的记忆库，而是做一个用户可以共同维护的“第二大脑”。
